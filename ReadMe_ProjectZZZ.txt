1. ЦЕЛЬ И БИЗНЕС-ПРАВИЛА
==============================================================================
Цель: Формирование персонализированного предложения из 5 SKU для каждого клиента на плановый визит менеджера (цикл: 1 визит в 14 дней).

Состав предложения (Правило 2+2+1):
[1] 2 SKU — Новые: клиент никогда не покупал (целевая вероятность > 0.8).
[2] 2 SKU — На развитие: из привычных, принадлежат маркетинговым группам с положительным трендом (тренд > +0.05).
[3] 1 SKU — На возврат: из привычных, принадлежат группам с отрицательным трендом (тренд < -0.05).

Критические условия:
- Окно покупки: [День визита (День 0), День 0 + 10 дней].
- Минимальная партия: 1 шт.
- Режим: Пакетная генерация ежедневно в 06:00.
- Выход: Защищенный Excel-файл.

2.3. Контроллер моделей (model_controller.py)
Функции:
1. Загрузка реестра моделей
2. Проверка статуса каждой модели (production/staging/disabled)
3. Сравнение метрик staging vs production
4. Автоматическое переключение если:
   - precision_5 улучшился на >= 3%
   - hit_rate улучшился на >= 3%
   - brier_score не ухудшился
   - training_time не увеличился более чем в 2 раза
5. Критический алерт если:
   - precision_5 упал на >= 5% (ТРЕБУЕТ НЕМЕДЛЕННОЙ РЕАКЦИИ)
   - hit_rate упал на >= 5% (ТРЕБУЕТ НЕМЕДЛЕННОЙ РЕАКЦИИ)
   - brier_score ухудшился на >= 10%

2.4. Ядро (MVP - обязательно)
- Алгоритм: LightGBM (Binary Classification).
- Задача: P(buy | client_id, sku_id, context) в окне 10 дней.
- Калибровка: sklearn.calibration.CalibratedClassifierCV с методом sigmoid (Platt scaling) на отложенной выборке (не обучающей). Это обеспечивает хорошую калибровку вероятностей для LightGBM.
- Метрика качества вероятностей: Brier Score.

2.5. Признаки (Features) для LightGBM
[A] Клиентские: frequency_30d, frequency_90d (частота покупок), avg_check_trend (динамика среднего чека), seasonality_index (месяц/квартал), portfolio_diversity (уникальных категорий за 6 мес)
[B] Товарные: global_popularity (частота покупок всеми клиентами), margin_coeff (нормализованная маржинальность), stock_level (остаток как признак, не только фильтр), is_new_flag (новинка каталога < 30 дней)
[C] Динамические (контекст визита): days_since_last_purchase_sku (для данного SKU), rolling_sales_2w, 4w, 8w (скользящие средние), cannibalization_score (корреляция с другими покупками клиента)
[D] Групповые (маркетинговые группы/бренды): group_trend_6m (линейный тренд покупок группы; для групп с менее чем 3 покупками за период тренд считать равным 0 – стабильные), group_share_in_portfolio (доля группы в покупках клиента), days_since_last_purchase_in_group
[E] Эмбеддинги (ALS - библиотека implicit): client_vec_1..50, sku_vec_1..50 (рассчитываются отдельно, подаются как фичи)

3. РЕТРОСПЕКТИВНОЕ ОБУЧЕНИЕ (BACKTESTING)
==============================================================================

3.1. Цель backtesting
Поскольку история реальных рекомендаций отсутствует, система имитирует прошлые визиты и формирует гипотетические предложения для обучения.

3.2. Процесс backtesting
Шаг 1. Выбор исторического периода (например, последние 12 месяцев)
Шаг 2. Для каждого исторического визита клиента:
   - Определить дату визита (например, каждые 14 дней по истории активности)
   - Сформировать гипотетические 5 SKU по текущим правилам отбора
   - Проверить фактические покупки в окне [визит, визит+10 дней]
   - Присвоить таргет: 1 если куплен, 0 если не куплен
Шаг 3. Накопление размеченных примеров (клиент, SKU, таргет, признаки)
Шаг 4. Обучение модели на размеченных данных
Шаг 5. Валидация на holdout-периоде (последние 2 месяца истории)

3.3. Критерии качества backtesting
- Минимум 10,000 размеченных примеров (визит × SKU)
- Precision@5 на валидации >= 0.35 (порог для запуска MVP)
- Hit Rate на валидации >= 0.55 (порог для запуска MVP)
- Brier Score <= 0.20 (порог для калибровки)

3.4. Скрипт backtesting (backtest_engine.py)
- Запускается один раз перед первым обучением модели
- Генерирует файл training_dataset.csv с размеченными данными
- Сохраняет метрики в model_registry.json
- Требует ручного подтверждения перед переходом к продакшн-обучению

==============================================================================
4. ЛОГИКА ОТБОРА (POST-PROCESSING)
==============================================================================

Алгоритм формирования 5 SKU (приоритетный порядок):

Шаг 1. Предсказание
- Для всех кандидатов (остаток >= 1) рассчитать P(buy) через активную модель.
- Применить калибровку вероятностей.

Шаг 2. Скоринг с учетом бизнес-правил
- Базовый скор: P_calibrated * margin_coeff
- Для "Развития": скор *= (1 + 0.5 * group_trend)  # бонус за растущие группы
- Для "Возврата": скор *= (days_since_last / max_days)  # бонус за давность

Шаг 3. Отбор по правилу 2+2+1
1. Новые: 
   - Сначала пытаемся взять топ‑2 новых SKU по скору, у которых P_calibrated > 0.8.
   - Если набралось <2, берём топ‑2 по скору без порога (включая SKU с вероятностью ниже 0.8).
   - Если всё равно <2, добираем из привычных с высокой маржой (логируем fallback_reason).
2. Развитие: Топ-2 по скору из "привычных" + растущие группы.
3. Возврат: Топ-1 по скору из "привычных" + падающие группы.

Шаг 4. Механизм Fallback (если не хватает кандидатов)
- Если "Возврат" пуст → брать из "Стабильных" групп с максимальной давностью покупки.
- Если после всех действий общее число SKU <5, дополнить любыми доступными товарами с остатками (логируя ошибку).
- Все замены логировать в поле 'fallback_reason'.

Шаг 5. Финальные проверки
- Удалить дубликаты.
- Проверить остатки (актуализация на момент генерации).
- Гарантировать ровно 5 SKU в выдаче (если невозможно — записать клиента в лист "Errors").

- backtest_results (результаты ретроспективного тестирования)

5.3. Пайплайн обновления данных:
[Ежедневно, 05:00]
1. Загрузка новых продаж → таблица purchases.
2. Обновление остатков/цен → таблица skus.
3. Пересчет динамических признаков (скользящие средние, days_since).

[Ежедневно, 06:00] - ГЕНЕРАЦИЯ РЕКОМЕНДАЦИЙ
1. Загрузка списка клиентов с визитами на сегодня.
2. Model Controller загружает активную модель из реестра.
3. Для каждого клиента:
   - Загрузка актуальных фич из PostgreSQL.
   - Предсказание вероятностей (активная модель + калибровка).
   - Применение логики отбора (п. 4) с учетом флагов моделей из конфига.
   - Сохранение результатов в visit_proposals.
4. Формирование и выгрузка Excel-файла.

[Ежедневно, 20:00] - ОБРАТНАЯ СВЯЗЬ
1. Для завершенных окон [Visit_Date, Visit_Date+10]:
   - Проверка фактических покупок из purchases.
   - Обновление флага was_recommended и таргета (1/0) в visit_proposals.
2. Накопление данных для следующего переобучения.

[Еженедельно, Воскресенье]
1. Пересчет трендов маркетинговых групп (с учётом правила для групп с <3 покупками: тренд = 0).
2. Запуск скрипта анализа A/B теста (если активен).
3. Обновление ALS эмбеддингов (опционально).
4. Проверка метрик активной модели по накопленным обратным связям за неделю:
   - precision_5 (доля купленных SKU среди предложенных)
   - hit_rate (доля визитов с хотя бы одной покупкой из предложения)
   - brier_score (качество калибровки)
   При деградации любой из метрик на >=5% — критический алерт с предложением отката.

[Ежемесячно, 1-е число]
1. Полное переобучение активной модели на скользящем окне (12 месяцев).
2. Валидация калибровки (Brier Score).
3. Тестирование staging-моделей на исторических данных.
4. Архивация старых прогнозов.

7. A/B ТЕСТИРОВАНИЕ И АВТОМАТИЗАЦИЯ РЕШЕНИЙ

7.1. Конфигурация A/B теста (D:\ProjectZZZ\config\config.yaml):
ab_test:
  enabled: true
  start_date: "2024-04-01"
  test_group_ratio: 0.5
  assignment_method: "hash"
  metrics:
    primary: "hit_rate_5"
    secondary: ["precision_5", "new_sku_adoption", "avg_check_uplift"]
  promotion:
    min_duration_days: 14
    significance_level: 0.05
    min_uplift: 0.03
    auto_promote: true
  degradation:
    critical_threshold: 0.05
    action: "rollback"

7.2. Логика распределения клиентов:
Детерминированное распределение: один клиент всегда в одной группе через hash(client_id) % 100

7.3. Автоматический анализ и переключение (скрипт evaluate_ab.py):
- Запускается еженедельно, если ab_test.enabled == true.
- Считает метрики для групп control/test за последние N дней.
- Проверяет статистическую значимость (t-test / z-test).
- Если условия выполнены (p-value < 0.05 AND uplift > 3%):
  * Если auto_promote: true → обновляет model_registry.json: переводит staging-модель в status: "production".
  * Отправляет уведомление разработчику (лог/почта).
- Если обнаружена деградация >= 5%:
  * Немедленный откат на предыдущую production-версию.
  * Отправка критического алерта разработчику.

7.4. Ручное управление:
- В любой момент можно установить auto_promote: false или enabled: false.
- Все изменения реестра логируются с указанием пользователя и времени.

8. МОНИТОРИНГ И КРИТЕРИИ ЭФФЕКТИВНОСТИ
==============================================================================

8.1. Технические метрики (ежедневный дашборд):
- pipeline_duration: время выполнения < 30 мин.
- coverage_rate: % клиентов с полным набором (5 SKU) > 95%.
- calibration_error: |P_pred - P_actual| < 0.1 в среднем.
- model_inference_time: < 100 мс на клиента.
- training_time: время обучения модели (сигнал при росте > 2х).
- error_rate: % клиентов в листе "Errors" < 5%.

8.2. Бизнес-метрики (для A/B тестов):
- Hit Rate @ 5: доля визитов с >=1 покупкой из рекомендованных.
- Precision @ 5: доля рекомендованных SKU, которые были куплены.
- New SKU Adoption: % новых артикулов в чеке клиентов тест-группы.
- Revenue Uplift: рост среднего чека / выручки на визит.

8.3. Пороги деградации (КРИТИЧЕСКИЕ)
[Красный уровень - требует немедленной реакции]
- precision_5 упал на >= 5% от базового значения
- hit_rate упал на >= 5% от базового значения
- brier_score ухудшился на >= 10%
- coverage_rate упал ниже 80%

Действия при красном уровне:
1. Автоматический откат на предыдущую стабильную версию модели.
2. Отправка алерта разработчику (email/telegram).
3. Блокировка авто-переключений до ручного разбора.

[Желтый уровень - требует наблюдения]
- Метрики не растут 3+ недели подряд.
- Время обучения увеличилось на 50-100%.
- Количество fallback-замен превысило 20%.

Действия при желтом уровне:
1. Логирование в дашборд.
2. Еженедельный отчет разработчику.
3. План оптимизации на следующий спринт.

8.4. Критерии принятия решений о новых моделях
[Экономические сигналы]
- Добавление новой модели увеличивает время обучения > 2х, а прирост метрик < 1%.
- Потребление памяти/диска выходит за пределы бюджета инфраструктуры.
- Сложность отладки: время на поиск бага > 4 часов при низком влиянии на бизнес.

[Правило принятия решения]
ЕСЛИ (простая модель дает > 85% от теоретического максимума) И (сложная модель дает прирост < 2% при увеличении стоимости в 3х) ТОГДА → не внедрять сложную модель, оптимизировать простую.

ЕСЛИ (новая модель дает прирост >= 3% по ключевой метрике) И (время обучения не увеличилось > 2х) И (стабильность подтверждена 2+ неделями теста) ТОГДА → перевести в production через model_registry.

12. КРИТИЧЕСКИЕ ДОПУЩЕНИЯ И РИСКИ
==============================================================================

[Допущения]
1. Менеджеры используют рекомендации без модификации (файл только для чтения).
2. Данные о продажах поступают ежедневно без задержек > 24 часов.
3. Маркетинговые группы и их иерархия стабильны (изменения реже 1 раза в квартал).
4. 26 месяцев истории достаточно для качественного backtesting.
5. PostgreSQL доступен 24/7 (локально или в сети).

[Риски и митигация]
1. Низкий Coverage (< 80%): → Снизить порог вероятности, упростить правило 2+2+1, добавить больше "стабильных" кандидатов.
2. Плохая калибровка (модель "врет" о вероятностях): → Увеличить размер валидационной выборки, попробовать другой метод калибровки.
3. Дрейф данных (сезонность, изменение ассортимента): → Использовать скользящее окно обучения, мониторить Brier Score по неделям.
4. Критическая деградация метрик (>= 5%): → Автоматический откат на предыдущую версию, алерт разработчику.
5. Сопротивление менеджеров: → Включить в Excel колонку "Объяснение" (почему рекомендован этот SKU), проводить обучение, собирать обратную связь.
6. Ошибки генерации для отдельных клиентов: → Клиент попадает в лист "Errors", файл всё равно создаётся для остальных. → Ошибка логируется, рекомендуется повторная попытка на следующий день.

14. ПРИЛОЖЕНИЕ: СЛОВАРЬ ТЕРМИНОВ
==============================================================================
- SKU (Stock Keeping Unit): уникальная товарная позиция (артикул).
- Backtesting: ретроспективное тестирование модели на исторических данных.
- Holdout выборка: отложенная часть данных для валидации (не используется при обучении).
- Калибровка вероятностей: приведение выходных значений модели к реальным частотам событий.
- Fallback: резервная логика отбора при невозможности выполнить основное правило.
- Coverage: доля клиентов, для которых удалось сформировать полное предложение.
- Precision@5: доля рекомендованных SKU (из 5), которые были фактически куплены.
- Hit Rate@5: доля визитов, где клиент купил хотя бы 1 рекомендованный SKU.
- Brier Score: метрика точности вероятностных прогнозов (чем меньше, тем лучше).
- A/B тест: сравнение двух версий системы на разных группах клиентов.
